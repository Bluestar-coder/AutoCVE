## V2EX / 开源中国短帖

标题：

```text
做了一个一键 CVE 挖掘 Agent：自动筛项目、审计源码、生成报告
```

正文：

```markdown
最近在做一个开源项目 AutoCVE，想把 CVE 挖掘里比较重复的流程自动化。

大概流程是：

输入想要的 CVE 候选数量 -> 自动从 GitHub 筛项目 -> 导入仓库 -> 创建 Agent 审计任务 -> Finding Agent 读代码挖漏洞 -> 漏洞管理 -> 生成中英文报告和 CVE 提交辅助内容。

它不是“保证拿 CVE”的工具，更准确地说是 CVE candidate discovery + evidence collection + report drafting。

技术上做了几块：

- Multi-Agent 审计流程：Orchestrator / Recon / Scan / Triage / Finding / Verification
- Finding Agent ReAct loop
- Continue / Terminal 状态转换
- Nudge 机制，避免模型只用自然语言假装完成
- ToolRegistry / ToolOrchestrator 工具编排
- Skill 机制，按需加载专项审计知识
- 漏洞管理和报告生成

目前比较适合安全研究、代码审计、AI Agent 工程化方向的朋友试用或拍砖。
```

## 掘金短帖

标题：

```text
我做了一个一键 CVE 挖掘 Agent，顺便踩了一遍安全 Agent 工程化的坑
```

正文：

```markdown
AutoCVE 是我最近在做的一个开源项目，目标是把 CVE 挖掘从“手动找项目、导仓库、跑任务、整理报告”变成一条可追踪的自动化流水线。

核心体验：

用户输入目标数量，比如 5 个 CVE 候选，系统会自动筛选 GitHub 项目，导入仓库，启动 Agent 审计，沉淀漏洞，并生成可以复制到 GitHub Advisory / CVE 提交流程里的报告内容。

工程上比较有意思的点：

1. Finding Agent 不是单次 prompt，而是 ReAct loop。
2. 每轮会根据工具调用、终止动作、token budget、stop hook 等判断 Continue 或 Terminal。
3. 如果模型说“我完成了”但没有调用 FinalizeFinding，会触发 nudge。
4. 工具调用统一走 ToolOrchestrator，做输入校验、权限检查、checkpoint 和执行记录。
5. Skill 机制用于按需加载审计方法、checklist 和报告流程，避免把所有知识塞进 prompt。

这项目不是想替代安全研究员，而是把重复流程交给 Agent，把漏洞判断和披露决策留给人。

项目地址：<GitHub 仓库链接>
```

## 先知 / FreeBuf / 安全客短帖

标题：

```text
AutoCVE：面向 CVE 挖掘的一键化 Agent 审计平台
```

正文：

```markdown
分享一个最近在做的开源项目 AutoCVE。

它面向代码安全审计和 CVE 候选挖掘，核心流程是：

GitHub 项目筛选 -> 仓库导入 -> Agent 审计 -> 漏洞管理 -> 报告生成 -> 人工复核后披露。

一键 CVE 批次任务会根据 star、近期活跃度、security advisory、SECURITY.md、private vulnerability reporting 等信号筛选项目，然后自动创建审计任务。

审计侧使用 Multi-Agent 流程：

- Recon：项目结构和攻击面识别
- Scan：自动化扫描
- Triage：误报过滤和证据补充
- Finding：基于 ReAct loop 的源码深挖
- Verification：沙箱验证和 PoC 辅助

Finding Agent 最终需要通过 FinalizeFinding 提交结构化漏洞结果，包括漏洞类型、文件路径、行号、source/sink、利用链、PoC、影响和 CVE 价值说明。

报告侧支持中文报告、English Report、CVE 报告，可用于后续 GitHub Advisory 或 CNA 提交前的材料整理。

强调一下：项目仅用于授权安全研究和学习。自动生成的漏洞结果需要人工复核，不建议未经确认直接提交。

项目地址：<GitHub 仓库链接>
```
