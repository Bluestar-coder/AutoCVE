# AutoCVE：从项目筛选到 CVE 报告产出，把漏洞挖掘流程自动化

> 适合平台：掘金、知乎、V2EX、公众号、开源中国、GitHub Release 附文  
> 建议标题：我做了一个一键 CVE 挖掘 Agent：输入目标数量，自动筛项目、挖漏洞、生成报告  
> 建议配图：一键 CVE 页面动图、Agent 审计日志、漏洞报告弹窗、架构图

## 开头

过去一段时间，我一直在做一个面向代码安全审计和 CVE 挖掘的开源项目：**AutoCVE**。

它想解决的问题很直接：安全研究员在挖 CVE 时，很多时间并不花在真正判断漏洞上，而是耗在一整条重复流程里：

- 找哪些开源项目值得看
- 判断项目是否活跃、是否有安全响应入口
- 导入仓库、配置模型、创建审计任务
- 在工具扫描和人工阅读之间反复切换
- 整理漏洞证据、影响范围、PoC、修复建议
- 最后再把内容改写成 GitHub Advisory、CNA 或 CVE 提交需要的格式

AutoCVE 的目标是把这条链路尽量产品化和自动化。

你可以把它理解成一个面向 CVE 挖掘场景的 Agent 审计平台：**从 GitHub 项目筛选、仓库导入、Agent 审计、漏洞管理，到中英文报告和 CVE 提交辅助内容生成，尽量串成一条可追踪的流水线。**

## 一句话介绍

AutoCVE 是一个一键 CVE 挖掘 Agent 平台。

用户输入想要的 CVE 候选数量后，系统会自动筛选 GitHub 项目，导入仓库，启动 Agent 审计，沉淀漏洞结果，并生成可复制的报告内容。

当然，它不是“保证拿 CVE”的魔法按钮。更准确地说，它是一个 **CVE 候选发现和报告生成辅助系统**：它帮助你更快发现高价值漏洞、更完整沉淀证据、更高效整理提交材料，最终仍然需要安全研究员人工复核和负责任披露。

## 它具体能做什么

### 1. 一键 CVE 批次任务

AutoCVE 里有一个专门的一键 CVE 入口。用户只需要输入目标数量，比如 3 个、5 个或 10 个候选漏洞，系统就会创建一个批次任务。

这个批次任务会自动做几件事：

1. 从 GitHub 搜索近期活跃、star 数较高的项目。
2. 根据项目活跃度、安全公告、SECURITY.md、private vulnerability reporting 等信号进行排序。
3. 自动导入候选仓库。
4. 为每个候选项目创建 Agent 审计任务。
5. 持续统计有效 finding，直到达到目标数量，或者候选项目耗尽。

这一步解决的是 CVE 挖掘里非常耗时间的“选项目”和“重复创建任务”问题。

### 2. Multi-Agent 审计流程

AutoCVE 不是简单地把一次模型调用包装成“AI 扫描器”。

它的审计流程里有多个 Agent 节点：

- Orchestrator：负责任务编排和结果合并
- Recon：识别项目结构、框架、入口文件和攻击面
- Scan：调用规则扫描、依赖扫描、密钥扫描等工具
- Triage：过滤误报、补充证据、判断候选有效性
- Finding：直接阅读源码，深挖高价值漏洞
- Verification：使用沙箱、测试工具和 PoC 做动态验证

用户可以根据目标切换不同模式。例如只想快速做工具扫描，可以走 Scan + Triage；如果目标是 CVE 或 0day 挖掘，则重点使用 Finding Agent。

### 3. Finding Agent：直接读代码的漏洞挖掘核心

项目里最核心的部分是 Finding Agent。

它不是读一遍扫描器输出然后总结，而是围绕源码进行多轮 ReAct loop：

1. 读取项目上下文、历史会话、Skills 和上游 handoff。
2. 调用模型生成原生工具调用。
3. 通过 Read、Grep、Glob、Shell、Skill 等工具继续查证。
4. 把工具结果写回会话。
5. 根据状态机判断继续、纠偏或终止。
6. 最终必须通过 `FinalizeFinding` 提交结构化漏洞结果。

这意味着每个漏洞结果都尽量包含：

- 漏洞类型
- 严重等级
- 文件路径和行号
- 关键代码片段
- source/sink
- 利用链
- PoC
- 影响说明
- 修复建议
- CVE 价值说明
- 验证备注

如果模型只是自然语言说“我完成了”，但没有提交结构化终止工具，runtime 会触发 nudge，让它继续查证或调用终止工具。这是为了避免很多 Agent 系统常见的“嘴上完成，状态没完成”问题。

### 4. 漏洞管理和报告生成

审计完成后，漏洞会统一进入漏洞管理模块。

这里可以做人工研判、修改状态、维护 CVE 申请进度，并查看报告。报告默认包含三类：

- 中文报告
- English Report
- CVE 报告

报告内容会尽量整理成提交友好的 Markdown，包括 Summary、Details、PoC、Impact、Remediation、Affected products、CWE、CVSS、Suggested CVE description 等字段。

如果你通过 GitHub Advisory 向项目维护者报告漏洞，很多内容可以直接复制过去，再做人工确认和润色。

### 5. Skill 机制

AutoCVE 还引入了 Skill 机制。

不同 Agent 可以绑定不同 Skill，比如：

- Java 反序列化审计
- SSRF 检测方法
- PHP 文件上传审计
- CVE 报告撰写
- huntr 提交流程
- 企业内部安全规范

Skill 不是一次性塞进 prompt 的长文本，而是采用渐进式加载：启动时先注入 Skill catalog 和 route plan，真正命中任务时再通过 `Skill(action="body")` 阅读完整说明，需要时继续读取 references、checklists、examples 或 scripts。

这样可以在控制上下文长度的同时，让 Agent 在具体任务里获得更专业的流程和知识。

## 为什么做这个项目

我做 AutoCVE 的原因很简单：漏洞挖掘不是只有“发现漏洞”这一个动作。

真正的 CVE 工作流更像是一条链：

```text
选项目 -> 导入仓库 -> 识别攻击面 -> 阅读源码 -> 查证漏洞 -> 验证影响 -> 整理报告 -> 跟踪提交状态
```

其中很多步骤都适合自动化，但又不能完全黑盒化。因为安全研究需要证据，需要可复核，需要谨慎处理披露边界。

所以 AutoCVE 的设计重点不是“让模型随便跑”，而是：

- 每一步有事件和日志
- 工具调用可追踪
- 会话可恢复
- 漏洞结果结构化
- 终止动作受约束
- 高风险工具有权限边界
- 报告可以人工编辑

它更像是给安全研究员配了一个能跑流程、能读代码、能整理材料的助手。

## 当前适合谁使用

AutoCVE 目前比较适合：

- 安全研究员
- 代码审计人员
- Bug bounty / SRC 玩家
- 对 AI Agent 安全审计感兴趣的工程师
- 想研究 ReAct loop、工具编排、Skill runtime 的开发者

如果你只是想要一个传统 SAST 工具，它可能不是最轻量的选择。

如果你想研究“如何把 Agent 真正接入复杂安全工作流”，AutoCVE 会比较有参考价值。

## 安全和合规说明

这个项目只应用于授权环境、安全研究和学习。

一键 CVE 不是鼓励无授权测试，也不是让用户批量骚扰开源维护者。任何自动发现的漏洞都应该经过人工复核，确认影响、版本、复现步骤和披露对象后，再按照项目的 SECURITY.md、GitHub private vulnerability reporting、CNA 或相关平台流程进行负责任披露。
